# Folderss 로컬 아이템 관리

현재 개발 아이템과 상태는 `docs/items/`의 항목별 Markdown 파일에서만 관리한다.
GitHub Project 연결이나 동기화는 개발 아이템 관리에 사용하지 않는다.

## 상태

| 상태 | 기준 |
|---|---|
| Todo | 요청 접수 후 아직 착수하지 않음 |
| In Progress | 현재 분석·설계·구현 중 |
| Ready for Verification | 구현과 자체 검증 완료, 사용자 최종 확인 대기 |
| Done | 사용자가 결과를 명시적으로 확인함 |

## 항목 파일

- 새 요청은 `docs/items/<항목>.md` 파일로 생성한다.
- 파일명은 영문 소문자와 하이픈을 우선 사용해 항목 내용을 짧게 표현한다.
- 요구사항, 원인 분석 또는 설계, 구현 내용, 변경 파일, 검증, 변경 이력을 항목 파일 본문에 직접 기록한다.
- 상태와 사용자 확인 여부도 항목 파일 안에서 관리한다.
- `docs/done/DONE.md`는 기존 릴리스 이력이며 현재 아이템 상태 인덱스로 사용하지 않는다.

## 필수 형식

```markdown
# <작업 제목>

- 상태: Todo | In Progress | Ready for Verification | Done

## 요구사항
## 원인 분석 또는 설계
## 구현 내용
## 변경 파일
## 검증
## 변경 이력
```

## 운용 원칙

- 요청 접수 시 항목 파일을 만들고 `상태: Todo`로 기록한다.
- 개발 착수 시 `상태: In Progress`로 변경한다.
- 구현과 자체 검증이 끝나면 `상태: Ready for Verification`으로 변경한다.
- 사용자가 "완료", "확인", "문제없음" 등으로 최종 결과를 승인한 경우에만 `상태: Done`으로 변경한다.
- 코드 작성과 빌드 성공만으로 `Done` 처리하지 않는다.
- 사용자의 추가 수정 요청은 같은 항목 파일에 요구사항, 구현 내용, 검증, 변경 이력을 누적 기록한다.
- 여러 버그나 기능이 독립적으로 확인될 수 있으면 각각 별도 항목 파일로 분리한다.
